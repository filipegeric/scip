Structured Concurrency

Every concurrent task's lifetime is bounded by a scope — the way variables are bounded by a block.

Lifetime

Tasks can't outlive the scope that spawned them

Errors

Failures propagate up to the parent scope

Cancellation

Cancelling a scope cancels all its children

Demo

The price each pays

CoroutineScope

  • Extra library (coroutines)
  • runBlocking + Dispatchers.IO
  • runInterruptible on every blocking call

StructuredTaskScope

  • JDK 25 preview, no extra libraries
  • Explicit join()
  • Helper wrapper for ergonomics

Which to pick?

CoroutineScope

StructuredTaskScope

Pick if

  • Already non-blocking / reactive
  • Need flexible cancellation patterns
  • Blocking JDBC / synchronous I/O everywhere
  • Virtual threads on the table
  • Readable stack traces matter

Avoid if

  • Mostly blocking codebase
  • Allergic to suspend
  • Preview APIs are a no-go

Fetching a book

Two I/O calls, one after the other. Total: ~2s.

First attempt

Compiles, misbehaves at runtime — webmvc + blocking JDBC weren't built for coroutines.

Try runBlocking

Works. Still ~2s — async starts eagerly on runBlocking's single thread; with no suspension points, the calls run back-to-back.

Dispatchers.IO

~1s. Happy path done.

What if reviews fails?

Book bumped to 3s to make the effect visible. Reviews fails after 1s, but we still wait 2 more seconds for the book. Worse than an error: a slow error.

runInterruptible

Blocking JDBC can't be cancelled — only interrupted. Now fails in 1s.

Just to make 2 calls in parallel

  • Extra library (coroutines)
  • runBlocking at the boundary
  • Dispatchers.IO for actual parallelism
  • runInterruptible on every blocking call

Coroutines are all-in

Mark one function suspend — mark every caller. The keyword spreads.
